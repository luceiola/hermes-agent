# Hermes 开发规范

## 1. 核心架构原则
- 原则名称：`主Agent决策，子Agent执行`（Intent-First Orchestration）。
- 定义：主 Agent 负责“是否执行、执行什么”；子 Agent/工具负责“如何执行”。

## 2. 输入与触发规则
- `输入 != 任务`：收到文本、图片、文件不等于自动触发业务流程。
- 所有外部输入必须先经过主 Agent 的意图识别与权限校验。
- 未收到明确指令前，默认行为是“确认已接收 + 等待用户指令”。
- 只有显式指令才允许触发子 Agent 或工具调用。

## 3. 职责边界
- 主 Agent 职责
1. 会话状态管理（等待指令、执行中、完成、取消）。
2. 意图判定与路由编排。
3. 权限、风控、平台策略检查。
4. 结果聚合与用户回复。
- 子 Agent / 工具职责
1. 执行单一原子能力（OCR、批改、检索、生成等）。
2. 返回结构化结果与错误信息。
3. 不做跨会话决策，不做隐式自动触发。

## 4. 接口设计规范
- 所有工具接口必须具备明确输入参数，不允许依赖“猜测型上下文”。
- 工具输出必须结构化（JSON schema），包含至少以下字段：
1. `success`、`elapsed_ms`、`errors`。
2. 业务核心结果字段（如 `questions`、`summary`）。
3. 原始响应保留字段（如 `raw_payload`）用于审计与调试。

## 5. 默认安全策略
- 默认最小动作：不确定就不执行重操作。
- 默认可追踪：关键路由与调用需可日志追溯。
- 默认可撤销：执行中任务应支持取消或超时终止。
- 默认可降级：外部 API 失败时给出可理解失败原因，不影响主流程稳定性。

## 6. 评审检查清单（PR 必检）
- 是否存在“根据输入类型自动触发业务”的隐式行为？
- 主 Agent 与工具层职责是否清晰分离？
- 工具是否仅做执行、不做决策？
- 是否有明确的显式指令触发条件？
- 输出是否满足结构化与可追踪要求？

## 7. 本规范适用范围
- 适用于 Hermes 全部 gateway/profile/agent 开发。
- 新增能力（技能、插件、工具、子 Agent）必须遵循本规范。
